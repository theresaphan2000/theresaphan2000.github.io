Bee's Key's 

Github Source
https://github.com/theresaphan2000/airbrakes

Bee's Key's is a service learning project I did for my final year at Clark. It was really one of the first "official" projects I started, and though it is not perfect I still learned a lot from it. For this, I designed a basic mechanical keyboard from bare bone AVR chips, along with designing a Python application that made remapping the keymaps easier. 

A basic keyboard that is typically sold to the common audience is extremely limiting in its customization. Usually, the product is sold with a preset keymapping layout that either is completely unchangeable or can be altered to one’s preference but requires intermediate levels of programming in C, AVRDUDE firmware flashing, and at least a general understanding of MCUS and data charts. Because of this, most keyboard users have no access to customizations that
cater to their preferences and/or ways to efficiently boost their productivity.With that, the intentions in developing this project was to simplify the process of reprogramming and flashing firmware onto a keyboard. To do so, I decided to build a small 3x3 dynamic macro-pad with fully customizable keys that can both be static, or dynamically macro. Alongside that, I developed a basic GUI Python application that significantly simplified the process of making the
hex file and flashing the file onto the MCU itself. Consequently, the user then does not need to know C or Bash Script, how to navigate the terminal, or have to read through countless articles and accidently brick their MCU. 

Designing the Keyboard

Throughout the term, I had many goals and milestones I wanted to reach for this project, such as:
             • Successfully wiping the Arduino bootloader from the ATMEGA328p chip. 
             • Rewriting the firmware to fit for specifications for customized keyboard
             • Flashing firmware onto the ATMEGA328P chip. 
             • Prototyping a hardwired keyboard. 
             • Prototyping a GUI Python application that can allow users to change macros, make and
               compile the file, along with flashing. 
             • Custom PCB for the keyboard
Though some of these goasl were not compelete, I will go into more detail about the parts I was able to compelete. 

When first designing the keyboard, I wanted to use a commonly used MCU which was the ATEMGA328P. This was because it was commonly used in Arduino Uno chips. 

Step Process

Here is my process of research and designing the keyboard. 

Steps Details
1. Wiping the Arduino Bootloader                                                                                                                                                                                                                                             I didn't want to use Arduino because I wanted a better understanding of embedded systems as a whole, and because using an Arduino device meant that it would only emulate an HID rather than being one itself. This was rather difficult because it was the first time I've worked with a baremetal chip other than PIC processors. What I needed to do was use AVRdude to flash the USBaspLoader that allowed the MCU to be an HID. The first few times, I had difficulty making this work because I wasn't doing it in command line, and was doing this in the Ubuntu terminal instead which caused targetting errors. To wipe and reupload a new bootloader, I had to use the USBtiny pocket programmer to achieve this. 
2. Making a standalone working chip For this, I basically needed to remove the chip form the Arduino and make a standalone to reduce space on the keyboard. This was a simple process, as all I needed were a few capacitors and a 16MHz crystal oscillator.                                                           
3. Writing and flashing the firmware onto the chip For the firmware, I used the QMK Firmare open source files as this was a commonly used on many keyboards. I had to configure my firmware to the atmega328p and rewrite the keymapping for the board. Getting the firmware onto the board was a lot simplier than the bootloader, as I had more experience with flashing MCU's at this point. 
4. Switching the chips Though I intended to only make a macro pad, I wanted to use a chip with pins in case I wanted to change the size of the keyboard or add more features such as LED lighting. I switched to the ATMEGA32A later on. The flashing and wiring was pretty similar because it was still an ATMEL chip. 
5. Making the application To make the application I used Python and its Tkinter libraries. This was a fairly straightforward process for programming the GUI application.
